# Tidymark - 产品需求文档（PRD）

## 1. 产品概述

**Tidymark** 是一款 Chrome 浏览器扩展，帮助用户整理长期积累的书签，解决“收藏了很多网站，但后续找不到、用不上、忘记了”的问题。

产品的核心价值有三点：
- 降低书签整理成本，让用户可以用 AI 快速完成分类与重组
- 提升书签找回效率，让用户能用自然语言搜索历史收藏
- 帮助用户识别“收藏后很少再用”的书签，便于清理和维护

## 2. 目标用户

重度书签使用者，拥有较多历史收藏，长期缺乏整理，且经常遇到以下问题：
- 记得收藏过，但找不到具体在哪个文件夹
- 收藏数量持续增长，但目录结构混乱
- 无法判断哪些书签已经长期闲置，值得清理

## 3. 产品目标与范围

### 3.1 产品目标

- 帮助用户对选定范围内的书签进行 AI 重组和整理
- 帮助用户通过自然语言快速找回已收藏书签
- 帮助用户识别长期不常访问或收藏后长期未再使用的书签

### 3.2 MVP 范围

MVP 包含以下能力：
- AI 智能分类
- AI 语义搜索
- 冷门书签识别与清理
- 本地数据保存、导入、导出、回滚

### 3.3 非目标

当前版本暂不包含：
- 自动定时重组书签
- 云端同步服务
- 多人协作
- 自动抓取网页正文并做深度内容理解

### 3.4 界面形态基调

MVP 的界面形态按“轻入口 + 重页面 + 侧边搜索”设计：
- `popup` 仅作为主入口，承载功能导航和状态摘要
- `side panel` 用于 AI 语义搜索，支持用户在浏览网页时持续搜索书签
- 完整页面用于复杂交互，包括分类结果编辑页和冷门书签页
- `options page` 用于设置、导入导出和快照管理
- 高风险操作统一通过确认弹层执行

## 4. 核心功能

### 4.1 AI 智能分类

| 项目 | 说明 |
|------|------|
| 触发方式 | 用户手动触发 |
| 作用范围 | 用户主动选择分类范围，可选择单个书签文件夹、多个文件夹，或全部书签 |
| 输入 | 所选范围内的书签标题、URL，以及现有文件夹结构信息 |
| AI 处理 | LLM 先生成建议分类结构，再将书签归入对应分类，可包含子分类 |
| 输出 | 分类结果编辑页 |
| 用户操作 | 审阅 AI 分类结果并直接修改（拖拽、重命名、合并分类、调整子分类）→ 确认后应用到所选范围 |

**补充说明：**
- 默认不自动改动任何书签，必须由用户手动发起
- 分类结果只作用于用户选定范围，不默认改动全部书签
- MVP 支持最多两级分类结构，即一级分类和一级分类下的子分类
- AI 可以生成一级分类及其子分类，但暂不支持更深层级
- AI 生成完成后，直接进入同一编辑页，预览与修改不拆分为两个步骤
- 编辑页默认可编辑，用户无需额外切换到编辑模式
- 编辑页应支持“未分类/待确认”分组，承接 AI 无法稳定判断的书签
- 编辑页应支持对子分类进行新建、拖拽归类、提升为一级分类等基础操作
- 编辑页应支持“重新生成”，MVP 阶段默认对当前选定范围整体重跑
- 确认替换前需要有明确的二次确认提示，避免误操作
- 每次应用分类前，需要保存一份可回滚快照
- 替换操作应支持一键回滚到替换前的状态
- 若 AI 返回结果异常或分类中断，不应对原始书签结构产生部分写入

### 4.2 AI 语义搜索

| 项目 | 说明 |
|------|------|
| 搜索范围 | 仅限用户已收藏的书签 |
| 交互方式 | 用户在侧边栏中用自然语言描述需求，系统返回匹配的书签列表 |
| 检索策略 | 本地先做标题、URL、文件夹名粗筛，再将候选集交给 AI 做语义排序与解释 |
| 输出 | 匹配的书签列表 + 匹配理由 |
| 结果操作 | 支持点击打开书签、定位所在文件夹 |

**示例：**
- 用户输入：“我想找之前收藏的那个在线画流程图的工具”
- 系统返回 draw.io、Excalidraw 等候选结果，并说明其匹配原因

**补充说明：**
- AI 搜索默认承载在浏览器 `side panel` 中，而不是 popup 或独立全屏页面
- 该能力定位为“找书签”的侧边搜索工具，不扩展为通用聊天助手

### 4.3 冷门书签识别

| 项目 | 说明 |
|------|------|
| 目标 | 帮助用户识别“不经常使用”或“收藏后长期未再使用”的书签 |
| 展示方式 | 以列表方式展示冷门书签，支持排序、筛选与批量处理 |
| 核心数据 | 收藏时间（`dateAdded`）、最后使用时间（`dateLastUsed`） |
| 默认排序 | 先展示 `dateLastUsed` 缺失的书签，其后按 `dateLastUsed` 升序排列，越久未使用的排越前 |
| 特殊规则 | `dateLastUsed` 缺失视为“收藏后从未再次使用” |
| 用户操作 | 支持单个删除、批量删除、打开书签、跳转到所在文件夹 |

**补充说明：**
- 该功能的目标是帮助用户清理闲置书签，不是展示最近活动
- 用户应能快速筛出“收藏后从未再次使用”“超过 X 天未使用”的书签
- 筛选条件可包括：从未再次使用、超过 30 天未使用、超过 90 天未使用、超过 180 天未使用
- MVP 基于 Chrome `bookmarks` 可用字段实现，不依赖 `history` 数据
- MVP 不展示访问频次

## 5. AI 服务配置

| 项目 | 说明 |
|------|------|
| 接口格式 | OpenAI API 兼容格式 |
| 用户配置项 | API Endpoint、API Key、模型名称 |
| 服务端 | 无，完全由用户自行提供 AI 服务 |
| 数据出站说明 | 使用 AI 功能时，所选范围内的书签标题、URL 等必要数据会发送到用户配置的模型服务 |

**补充说明：**
- 首次使用 AI 功能前，需要明确告知用户将向其配置的第三方模型服务发送书签数据，并要求用户确认
- 非 AI 功能不依赖外部服务
- 插件本身不提供托管模型服务

## 6. 数据方案

| 项目 | 说明 |
|------|------|
| 存储位置 | 全部本地存储 |
| 云端服务 | 无 |
| 本地保存内容 | AI 服务配置、分类快照、冷门书签相关元数据、插件本地设置 |
| 导出 | 支持导出为单个 JSON 文件，包含插件配置、插件保存的书签元数据，以及导出时对应的书签目录结构 |
| 导入 | 支持在新机器上导入上述导出文件，恢复插件状态和对应书签目录结构 |

**补充说明：**
- 导出文件需要包含 `schemaVersion` 字段，用于后续版本兼容和迁移
- 导出文件建议包含以下顶层结构：`settings`、`snapshots`、`bookmarksTree`、`metadata`
- 导入恢复的范围包含书签目录结构，不仅是插件配置
- 导入操作属于高风险操作，需要在执行前明确提示
- 回滚快照与导入恢复都应以“恢复到某个明确状态”为目标，而不是做模糊合并
- 回滚快照默认保留最近 10 份，超过上限时自动删除最旧快照
- 用户可以手动删除旧快照
- 仅在高风险操作前生成快照，包括 AI 重组、导入恢复、批量删除

## 7. 性能考量

| 场景 | 策略 |
|------|------|
| 书签量大（数千条） | AI 分类采用分阶段处理，先生成分类骨架，再分批归类 |
| AI 搜索 | 本地先做粗筛，缩小候选集后再调用 AI |
| 高风险操作 | 重组、导入、回滚前先生成本地快照，成功后再提交最终变更 |

## 8. 权限与风险

| 项目 | 说明 |
|------|------|
| 高风险操作 | AI 重组、导入恢复、批量删除 |
| 风险控制 | 二次确认、结果预览、可回滚、明确提示影响范围 |
| 隐私风险 | 用户书签数据可能发送到其自行配置的模型服务 |
| 权限原则 | 仅申请完成功能所需的最小权限 |

## 9. 待确认事项

当前版本暂无阻塞性的待确认事项。

---

这版 PRD 已经明确了以下产品决策：
- AI 分类的重组范围由用户主动选择
- 导入/导出恢复范围包含书签目录结构
- AI 功能存在数据出站，需要在首次使用前明确告知并确认
- “时间线”功能已收敛为“冷门书签识别”，目标是帮助用户发现长期闲置的书签
- 界面形态采用 `popup` 入口、`side panel` 搜索、完整页面处理复杂任务的组合方式
- AI 分类在 MVP 阶段支持最多两级目录结构
