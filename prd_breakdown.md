# Tidymark PRD 拆分稿

## 1. 使用方式

本拆分稿用于把 [tidymark.md](/Users/lovegakki/workspace/myself/codespace/web/Tidymark/tidymark.md) 转成后续设计、排期和开发可以直接引用的执行文档。

输出分为三部分：
- 功能清单：定义 MVP 要做什么、每项能力的边界和验收重点
- 页面清单：定义用户会看到哪些界面、每个界面的核心模块和状态
- 技术实现草案：定义 MVP 推荐的扩展架构、数据模型和关键流程

## 2. 功能清单

### F1. AI 智能分类

**目标**
- 帮助用户对选定范围内的书签进行一次性 AI 重组和整理

**用户价值**
- 降低大规模书签整理成本
- 快速形成新的目录结构

**主链路**
- 用户选择分类范围
- 系统读取所选范围内的书签与目录结构
- AI 生成分类骨架并完成书签归类
- 用户进入分类结果编辑页进行手动调整
- 用户确认应用，系统先保存快照，再写入最终结构

**子能力**
- 选择作用范围：单个文件夹、多个文件夹、全部书签
- 生成分类结果
- 编辑分类结果：拖拽、重命名、合并分类、调整子分类
- 显示“未分类/待确认”分组
- 重新生成当前选定范围的结果
- 应用前二次确认
- 一键回滚到操作前快照

**边界**
- 默认不自动执行，必须由用户手动发起
- MVP 只支持最多两级分类结构
- 不支持定时自动分类
- 不支持仅重跑某个局部分类，MVP 只支持当前范围整体重跑

**验收重点**
- 未确认前不得修改原始书签结构
- AI 异常或中断时不得产生部分写入
- 应用分类前必须生成快照
- 分类结果只影响用户选择的范围
- AI 生成结果允许包含子分类，但层级不得超过两级

### F2. AI 语义搜索

**目标**
- 帮助用户通过自然语言快速找回已收藏书签

**用户价值**
- 降低“记得收藏过但找不到”的成本

**主链路**
- 用户输入自然语言需求
- 系统基于标题、URL、文件夹名做本地粗筛
- AI 对候选结果进行语义排序和解释
- 用户直接打开书签或定位到其所在文件夹

**子能力**
- 自然语言输入
- 候选结果列表
- 匹配理由展示
- 打开书签
- 跳转到所在文件夹

**边界**
- 搜索范围仅限已收藏书签
- 不做联网网页搜索
- 不做书签内容级索引

**验收重点**
- 查询结果需要有可解释的匹配理由
- 无结果时需要明确反馈
- AI 服务不可用时，需要给出可恢复的错误提示

### F3. 冷门书签识别与清理

**目标**
- 帮助用户发现长期未再次使用的书签，并进行清理

**用户价值**
- 持续控制书签规模
- 找出“收藏后再也没用过”的书签

**主链路**
- 系统读取书签的 `dateAdded` 与 `dateLastUsed`
- 根据规则生成冷门书签列表
- 用户按条件筛选和排序
- 用户选择打开、定位或删除书签
- 批量删除前保存快照并二次确认

**子能力**
- 默认排序：优先显示 `dateLastUsed` 缺失书签，再按最后使用时间升序
- 快速筛选：从未再次使用、超过 30/90/180 天未使用
- 单个删除
- 批量删除
- 打开书签
- 跳转到所在文件夹

**边界**
- MVP 不依赖 `chrome.history`
- MVP 不展示访问频次
- 该能力定位为清理工具，不是活动时间线

**验收重点**
- `dateLastUsed` 缺失统一定义为“收藏后从未再次使用”
- 删除操作纳入统一快照与确认流程

### F4. AI 服务配置

**目标**
- 允许用户接入自定义 OpenAI 兼容模型服务

**子能力**
- 配置 API Endpoint
- 配置 API Key
- 配置模型名称
- 首次使用 AI 前展示数据出站说明

**边界**
- 插件自身不提供托管模型服务
- 非 AI 功能不依赖外部服务

**验收重点**
- 未完成配置时，AI 功能应阻止继续并引导前往设置
- 首次使用前必须显式确认数据出站

### F5. 本地数据、导入导出与回滚

**目标**
- 为高风险操作提供恢复能力，并支持跨机器迁移

**子能力**
- 本地保存配置、快照、元数据、本地设置
- 导出为单个 JSON 文件
- JSON 中包含 `schemaVersion`
- 导入时恢复插件状态和对应书签目录结构
- 回滚最近 10 份快照
- 手动删除旧快照

**边界**
- 导入恢复以“恢复到明确状态”为目标，不做模糊合并
- 快照仅在高风险操作前生成：AI 重组、导入恢复、批量删除

**验收重点**
- 导入、回滚、批量删除都需要高风险提示
- 超过快照上限时自动删除最旧快照

## 3. 页面清单

### P1. Popup 首页 / 主控制台

**目标**
- 提供所有核心能力的统一入口

**入口**
- 点击扩展图标打开 `popup`

**核心模块**
- AI 智能分类入口
- 打开 AI 搜索侧边栏入口
- 冷门书签入口
- 最近快照摘要
- AI 配置状态提示

**页面定位**
- 只做导航和状态摘要，不承载复杂编辑操作

**关键状态**
- 未配置 AI
- 已配置 AI
- 当前无快照
- 当前有可回滚快照

**核心模块**

- 风险提示
- 开始生成按钮

**关键状态**

- 已选择单个文件夹

- 已选择全部书签

### P3. 分类结果编辑页（完整页面）

**目标**
- 承接 AI 生成结果，并允许用户直接调整后应用

**核心模块**
- 分类分组列表
- 子分类分组
- 每组内书签列表
- 未分类/待确认分组
- 拖拽交互
- 分类重命名
- 分类合并
- 子分类提升为一级分类
- 重新生成按钮
- 应用按钮

**关键状态**
- AI 生成中
- 生成成功
- 生成失败
- 有未分类项
- 有未保存修改

**高风险动作**
- 应用分类结果

### P4. AI 搜索侧边栏（Side Panel）

**目标**
- 提供自然语言检索入口和结果列表

**核心模块**

- 作为持续可见的侧边工具存在，不打断用户当前浏览页面
- 定位为“找书签”工具，不扩展为通用聊天窗口

**关键状态**

**目标**
- 集中展示长期未再次使用的书签并支持清理

**关键状态**

## 4. 技术实现草案

### 4.1 MVP 架构建议

**推荐模块**
- `popup`：首页与轻量入口
- `side panel`：AI 搜索
- 完整页面：分类编辑页、冷门书签页
- `options` 页面：设置、快照、导入导出
- `background/service worker`：统一调度 Chrome API、AI 请求、快照写入
- 本地存储层：持久化配置、快照、元数据和本地状态

**模块职责**
- UI 层只负责展示、收集用户输入和触发动作
- 后台层负责书签读写、AI 调用、快照和错误处理
- 存储层负责 schema 管理、快照上限控制、导入导出序列化

### 4.2 Chrome API 依赖

**核心依赖**
- `chrome.bookmarks`
- `chrome.storage`
- `chrome.downloads` 或等价导出方案

**可选依赖**
- 若导入文件采用用户本地选择方式，可通过页面文件选择实现，不必增加额外浏览权限

**当前不依赖**
- `chrome.history`

### 4.3 数据模型建议

**settings**
- `apiEndpoint`
- `apiKey`
- `model`
- `hasConfirmedOutboundNotice`

**metadata**
- `schemaVersion`
- `lastExportAt`
- `lastImportAt`
- `lastAiRunAt`

**snapshots**
- `id`
- `createdAt`
- `type`：`ai_reorganize`、`import_restore`、`bulk_delete`
- `scope`
- `bookmarksTree`
- `metadata`

**local ui state**
- 最近使用的筛选条件
- 最近一次 AI 分类的选定范围
- 搜索历史

### 4.4 AI 分类流程

1. 读取用户选定范围内的书签树
2. 生成标准化输入，只包含标题、URL、父级结构等必要字段
3. 先调用一次 AI 生成分类骨架
4. 再按批次把书签归入骨架分类
5. 将结果返回分类结果编辑页
6. 用户确认后，后台先写入快照
7. 快照成功后再执行书签树改写

**失败处理**
- AI 失败：保留当前原始结构，不写入书签
- 快照失败：阻止后续写入
- 书签写入失败：提示失败并保留最近快照以便手动恢复

### 4.5 AI 搜索流程

1. 用户输入自然语言
2. 本地基于标题、URL、文件夹名做候选粗筛
3. 将候选集与查询一起提交给 AI
4. AI 返回排序结果和简要理由
5. `side panel` 展示结果并支持打开、复制或定位

### 4.6 冷门书签计算流程

1. 读取全部书签节点
2. 过滤出 URL 类型节点
3. 读取 `dateAdded` 与 `dateLastUsed`
4. 标记 `dateLastUsed` 缺失项为“从未再次使用”
5. 生成默认排序与筛选索引
6. 用户执行删除前先创建快照

### 4.7 导入导出与回滚流程

**导出**
- 从本地存储读取 `settings`、`snapshots`、`metadata`
- 从当前书签系统读取 `bookmarksTree`
- 组装单个 JSON 文件并带上 `schemaVersion`

**导入**
- 读取 JSON
- 校验 `schemaVersion`
- 展示恢复范围与风险提示
- 确认后先生成当前状态快照
- 再恢复导入文件中的目标状态

**回滚**
- 用户选择目标快照
- 系统展示快照摘要
- 确认后恢复快照中的书签结构与对应元数据

### 4.8 错误处理与风控

**统一错误类型**
- AI 配置缺失
- AI 调用失败
- 本地快照写入失败
- 书签改写失败
- 导入文件格式不合法
- schema 版本不兼容

**统一风控要求**
- 高风险操作前必须弹出确认
- 高风险操作前必须尝试生成快照
- 错误提示需要明确告诉用户当前是否已修改系统状态

### 4.9 开发顺序建议

1. 基础扩展骨架与本地存储层
2. 书签读取、快照、导入导出基础能力
3. 冷门书签页
4. AI 配置页与首次出站确认
5. AI 搜索侧边栏
6. AI 分类范围选择与分类结果编辑页
7. 高风险确认弹层与统一错误处理
